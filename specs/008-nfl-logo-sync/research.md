## Finding 1: ESPN API Shape and Logo URL

- **Decision**: Use `GET https://site.api.espn.com/apis/site/v2/sports/football/nfl/teams?limit=1000`
- **Rationale**: Publicly documented in `docs/nfl-team-logo-sync-instructions.md`. Returns all 32 teams with `team.logos[0].href` pointing to a stable ESPN CDN PNG. No authentication required.
- **Alternatives considered**: Hotlinking the ESPN CDN URL directly — rejected because the spec requires local storage (FR-002, FR-007).
- **Reference**: `docs/nfl-team-logo-sync-instructions.md`

---

## Finding 2: Image Download Without New npm Packages

- **Decision**: Use Node's built-in `https.get()` to stream image bytes into a `fs.createWriteStream()` targeting `packages/frontend/public/logos/nfl/<abbr>.png`.
- **Rationale**: No additional npm packages are needed. The built-in streaming approach handles binary files efficiently and is idiomatic for Node 18+. `better-sqlite3` and `axios` are already present for their respective concerns.
- **Alternatives considered**: `axios` (already in frontend, not backend package); `node-fetch` (would add a new dependency) — both rejected to keep the dependency surface minimal.

---

## Finding 3: SQLite Database Bootstrap

- **Decision**: Create `packages/backend/src/services/nflTeamsStore.js` that opens/creates `packages/backend/data/capstone.db` via `better-sqlite3`, runs `CREATE TABLE IF NOT EXISTS nfl_teams (...)`, and exports query helpers (`upsertTeam`, `getAllTeams`). The `data/` directory is created programmatically if absent.
- **Rationale**: `better-sqlite3` is already listed in `packages/backend/package.json` but not yet used in any source file. This feature is its first active use. A single shared DB file (`capstone.db`) avoids multiple SQLite files and aligns with the monolithic backend design; future tables can be added to the same file.
- **Alternatives considered**: Keeping team data in the in-memory `dataStore.js` arrays — rejected because the spec explicitly requires a database table (FR-004) to support persistence across server restarts and to decouple the sync script from the running server process.

---

## Finding 4: Frontend Static Asset Serving

- **Decision**: Store logo PNGs at `packages/frontend/public/logos/nfl/<abbr>.png`. The browser accesses them at root-relative URL `/logos/nfl/min.png` — no backend route required.
- **Rationale**: Create React App serves everything under `public/` as static assets at the domain root. This matches FR-007 and the clarification session answer exactly.
- **Alternatives considered**: A backend route that streams the file on demand — rejected as unnecessary complexity when CRA static serving already handles it.

---

## Finding 5: Frontend Logo Lookup Strategy

- **Decision**: `DraftRoomPage` fetches `GET /api/nfl-teams` on mount and builds a `Map<abbreviation, logoPath>` in component state. `DraftPlayerRow` and `DraftFilters` receive the map as a prop and look up logos by `player.nflTeamAbbr` or filter value.
- **Rationale**: Avoids global state or Context changes for a concern scoped entirely to the Draft Room page tree. The map is lightweight (32 entries) and straightforward to pass via props.
- **Alternatives considered**: React Context for NFL teams — rejected as premature; no other page currently needs this data.

---

## Finding 6: Fallback for Missing Logos (NFR-002, SC-002)

- **Decision**: `DraftPlayerRow` renders `<img src={logoPath} alt="{team name} logo" onError={handleImgError} />` where `handleImgError` sets a boolean state flag that hides the `<img>` and shows the abbreviation text span instead. No broken-image icon is ever visible.
- **Rationale**: The `onError` pattern is the standard React approach for image load failures and requires no extra libraries. This satisfies NFR-002 (page must load correctly without logos) and SC-002 (no broken image icons).
